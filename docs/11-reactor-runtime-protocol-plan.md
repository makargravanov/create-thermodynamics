# План исполнительного слоя реакторов

## Цель

Сделать реактор полноценной серверной подсистемой, а не набором действий в сущностях блоков. Мультиблок должен определять форму и порты, JVM должна владеть состоянием мира и сохранением, Rust должен владеть расчётом химии активного реактора.

Главное правило: нативный реактор не является долговременным источником истины. Долговременным источником истины является сериализуемый снимок реактора и внешние операции, ожидающие подтверждения Rust. Нативный экземпляр можно создать, остановить и восстановить из снимка.

## Термины

- **Структура**: собранный мультиблок с `structureId`, камерой, контроллером и портами.
- **Снимок реактора**: сериализуемое химическое состояние зон, смесей, портовых буферов, настроек, метрик и версии.
- **Нативный экземпляр**: активный Rust-объект, созданный из снимка и выполняющий расчёты.
- **Команда**: запрос JVM к нативному исполнителю.
- **Отчёт**: событие от нативного исполнителя к JVM.
- **Подтверждённый снимок**: последний снимок, который JVM получила от Rust и успела применить в серверном потоке.
- **Ожидающая подтверждения внешняя операция**: намерение изменить химическое состояние из Minecraft-мира, которое ещё не подтверждено отчётом Rust. Например, предмет уже лежит во входном буфере порта, но ещё не списан как вещество, внесённое в смесь.

## Слои

```text
Minecraft server thread
  читает мир
  применяет отчёты
  сохраняет снимки
  меняет блоки, меню и предметные хранилища

JVM reactor coordinator
  собирает команды
  передаёт команды в JNI
  ждёт отчёты не в серверном потоке
  складывает готовые отчёты в очередь серверного потока

Rust reactor worker
  владеет активными реакторами
  применяет команды последовательно
  считает химию
  пишет отчёты в очередь
```

NeoForge-слой не должен содержать химию и не должен напрямую владеть нативными реакторами. Его задача: события мира, регистрация обработчиков, сетевые сообщения и тонкое применение результатов к Minecraft.

## Учёт существующей нативной структуры

В нативной части уже есть не только `ReactorsWorker`. Исполнительный слой должен опираться на текущие части модели, а не заменять их:

```text
MinecraftChemistryState
  статический каталог Destroy
  привязки предметов Minecraft к веществам
  ReactorsWorker

DynamicChemistryRegistry
  динамические вещества
  динамические реакции как пересобираемый производный слой
  кислотно-основные спецификации
  осаждение
  комплексы
  индексы реакций по веществам
```

Текущий `MinecraftChemistryState` пока строит обычный `ChemistryRegistry` через `destroy_registry_builder()`. Для рабочего реактора этого достаточно только до тех пор, пока смесь содержит исключительно статические вещества и статические реакции.

Для полноценной модели реакторов состояние нативного мира должно включать каталоговую часть:

```text
NativeMinecraftRuntime
  catalog: RuntimeChemistryCatalog
  itemBindings: MinecraftChemicalRegistry
  reactors: ReactorsWorker
  commandQueue
  reportQueue
```

`RuntimeChemistryCatalog` не должен быть новой параллельной химической моделью. Он должен быть обёрткой над существующим `DynamicChemistryRegistry` или проверяемым снимком, восстановленным из него.

Минимальное правило:

- если реактор содержит только статические вещества, снимку достаточно `SubstanceId`;
- если реактор содержит динамические вещества, снимок мира обязан содержать их канонические структуры и версию динамического каталога;
- если во время реакции или генерации появились новые динамические вещества, отчёт Rust должен вернуть не только изменение смеси, но и расширение каталога;
- JVM не должна сама вычислять динамические вещества, но должна сохранять данные, нужные Rust для восстановления каталога.

Иначе после перезагрузки мира возможна некорректная ситуация: смесь ссылается на `dynamic:*`, а восстановленный нативный каталог такого вещества ещё не знает.

## Источники истины

### Мировой источник истины

В `v1_21_1-common` нужен единый владелец состояния реакторов уровня, например:

```kotlin
class ReactorWorldRuntime {
    val structures: ReactorStructureWorldStore
    val reactors: ReactorRuntimeStore
    val catalog: ReactorCatalogRuntimeState
    val commandOutbox: ReactorCommandOutbox
    val reportInbox: ReactorReportInbox
}
```

Он должен хранить:

- `structureId -> ReactorSavedState`;
- `structureId -> ActiveNativeReactor?`;
- `runtimeCatalogVersion`;
- `dynamicCatalogSnapshot`;
- `blockPos -> ReactorBlockMembership`;
- `controllerPos -> structureId`;
- `portPos -> ReactorPortRuntimeState`;
- последние причины отказа сборки и нативных операций.

Сущности блоков хранят только локальную проекцию для отрисовки, открытия меню и маленький внешний буфер порта, если этот буфер является частью поведения конкретного блока. Они не решают, собрана структура или нет, и не являются владельцами химии.

### Нативный источник истины во время расчёта

Пока реактор активен, Rust владеет расчётным состоянием. JVM не пытается пересчитывать химию и не изменяет химический снимок напрямую. Любое изменение идёт командой, а фактом становится только отчёт Rust, применённый в серверном потоке.

## Нативные двоичные снимки

Общее правило для сериализации:

> Всё, что является химической моделью, реакционным пространством или производным расчётным состоянием Rust, сохраняется как сжатый двоичный нативный снимок. JVM хранит только ссылку, тип, версии, размеры и хэш.

Это относится не только к динамическому каталогу, но и к:

- динамическим веществам и молекулярным графам;
- спецификациям кислот, осадков, комплексов, катализаторов и поверхностей;
- крупным снимкам реакционного пространства;
- полным химическим снимкам реакторов;
- внутренним проверяемым ускорителям, если они когда-либо будут сохраняться.

Динамические реакции, каналы реакций и обработанные ключи генерации не являются долговременной истиной каталога, если они могут быть детерминированно пересобраны из веществ, графов, версий генераторов и текущего запроса. Их нельзя сохранять как список идентификаторов: такой список не содержит химии и создаёт второй источник истины.

Единая обёртка на стороне JVM:

```kotlin
data class NativeBlobRef(
    val kind: NativeBlobKind,
    val schemaVersion: Int,
    val modelVersion: String,
    val contentVersion: Long,
    val contentHash: ByteArray,
    val compressedByteSize: Long,
    val uncompressedByteSize: Long,
    val storageKey: String,
)
```

Пример видов:

```kotlin
enum class NativeBlobKind {
    DynamicCatalogCheckpoint,
    DynamicCatalogDelta,
    ReactorSnapshot,
    ReactionSpaceSnapshot,
    PlannerSnapshot,
}
```

JVM не должна иметь классы вида `DynamicReactionSnapshot`, если эти классы существуют только для сохранения и восстановления химии. Реакции, каналы и расчётные индексы должны быть секциями Rust-формата, а не объектной моделью Minecraft-слоя.

Человекочитаемые данные допустимы только для:

- состояния мультиблока;
- позиций и ролей блоков;
- настроек портов;
- маленьких внешних буферов портов;
- ссылок на нативные двоичные снимки;
- кратких сводок и страниц интерфейса, запрошенных у Rust;
- диагностик, которые нужны игроку или журналу.

Если JVM не должна принимать решения по содержимому данных, она не должна их распаковывать и раскладывать на Kotlin-объекты.

## Снимок реактора

Добавить сериализуемый Rust-формат. Это не означает, что JVM должна раскладывать его в объектную модель при сохранении мира:

```text
ReactorSnapshot {
  protocolVersion
  structureId
  snapshotVersion
  catalogVersion
  geometryHash
  zones[]
  transitions[]
  ports[]
  mixtures[]
  persistentContext
  outputBuffers
  metrics
  diagnostics
}
```

Снимок должен содержать достаточно данных, чтобы восстановить нативный экземпляр без чтения старого нативного объекта. В мировом сохранении JVM может хранить его как `NativeBlobRef(kind = ReactorSnapshot, ...)`, а Rust восстанавливает и проверяет содержимое.

Минимум для первой версии:

- версия протокола, версия снимка и версия каталога;
- `structureId`, проверочная сводка формы структуры и `geometryHash`;
- зоны: объём, режим объёма, герметичность, прошедшее расчётное время;
- смесь каждой зоны: температура, газовый объём и полный фазовый состав;
- переходы между зонами, входы и выходы: режим, список веществ или фаз, скорость, порог и признак включения;
- настройки входных и выходных портов;
- химические выходные буферы;
- постоянный контекст реакции, если он задан реактором: свет, газовая среда, поверхности, внешние катализаторы и внешние реагенты;
- настройки теплообмена с окружением, если они заданы;
- параметры равновесия фаз, если они изменяемы для конкретного реактора;
- последние метрики и диагностики в ограниченном кольцевом буфере.

### Состав смеси в снимке

Смесь нельзя сохранять как одно число концентрации на вещество. Нужно сохранять фактическое распределение по состояниям:

```text
MixtureComponentSnapshot {
  substanceId
  aqueousMolPerBucket
  organicMolPerBucketBySolventId
  moltenMolPerBucketByPhase
  gasMolPerBucket
  supercriticalMolPerBucket
  solidMolPerBucketByPhase
}
```

Ключи, которые в Rust сейчас являются индексами, в сохранении должны быть стабильными идентификаторами:

- `SubstanceIndex` сохранять как `SubstanceId`;
- растворитель органической фазы сохранять как `SubstanceId`;
- якорь твёрдой или расплавленной фазы сохранять как `SubstanceId`;
- внутренние позиции компонентов и индексы пересобирать при восстановлении.

Иначе после перезагрузки можно получить смесь с правильным общим количеством вещества, но с потерянной фазой. Для реактора, перегонки, газового зазора, горения и осаждения это будет уже другое физическое состояние.

### Что не является источником истины

Не сохранять как авторитетные данные:

- `ReactorInstanceId`: это идентификатор активного нативного экземпляра, а не состояние мира;
- внутренние индексы вещества, реакции, сайта, фазы и компонента;
- снимки экрана контроллера;
- расчётные таблицы доступных реакций для смеси, если их можно перестроить из каталога и смеси;
- полную историю тиков.

Эти данные можно кэшировать только как производные и сбрасываемые. После восстановления они должны пересобираться из снимка каталога, снимка реактора и структуры мира.

Запрещено восстанавливать реактор из полей сущностей блоков по частям. Сущности блоков могут содержать только копию для отображения и сохранения инвентаря порта.

## Жизненный цикл структуры

```text
NOT_FORMED
  нет валидной структуры

FORMED_NO_NATIVE
  структура валидна, но нативный экземпляр ещё не создан

ACTIVE_NATIVE
  структура валидна, нативный экземпляр создан и принимает команды

SUSPENDED_UNKNOWN
  часть нужных чанков не загружена; структуру нельзя считать сломанной

SUSPENDED_UNLOADED
  структура выгружена намеренно, есть сохранённый снимок

INVALID
  структура точно нарушена, причина известна

REMOVED
  структура удалена, нативный экземпляр удалён, состояние либо сохранено как остаток, либо уничтожено явно
```

Переходы должны быть явными и проверяемыми. Нельзя превращать неизвестное состояние чанков в разрушение структуры.

## Обмен командами и отчётами

### Базовый вариант

Начать с коротких JNI-вызовов пачками:

```text
nativeSubmitReactorCommands(batch) -> SubmitResult
nativePollReactorReports(cursor, maxReports, timeoutMillis) -> ReportBatch
nativeWakeReactorWorker()
nativeShutdownReactorWorker()
```

`nativePollReactorReports` может ждать отчёты, но не в серверном потоке. Серверный поток должен использовать только готовую JVM-очередь отчётов.

### Почему не обратные вызовы из Rust в JVM

Rust не должен вызывать Minecraft-код напрямую. Нативный поток не имеет права трогать мир, блоки, сущности, меню или игрока.

Правильный путь:

```text
Rust worker -> очередь отчётов -> JVM collector thread -> серверный поток Minecraft
```

Так нативный код может сообщать события сразу, но мир меняется только в серверном потоке.

### Долгий опрос

Долгий опрос допустим, если выполнены условия:

- вызов не выполняется в серверном потоке;
- есть ограничение ожидания, например 50-100 мс;
- есть `nativeWakeReactorWorker()` для немедленного пробуждения;
- есть `nativeShutdownReactorWorker()` для остановки;
- при закрытии мира ожидающий поток гарантированно выходит;
- нативный код не держит ссылки на объекты Minecraft.

Серверный поток не должен ждать нативу. Он только применяет уже готовые отчёты.

### Общая память

Обмен через выделенную вне управляемой кучи память можно добавить позже, но не как первый слой.

Условия для перехода:

- формат команд и отчётов стабилен;
- есть версия протокола;
- есть проверки повреждённых данных;
- есть ограничение размера пачки;
- есть тесты владения буфером;
- есть измерения, показывающие, что JNI-пачек недостаточно.

До этого общий участок памяти создаст больше рисков, чем пользы.

## Команды

Каждая команда должна иметь:

```text
ReactorCommand {
  commandId
  structureId
  expectedSnapshotVersion
  payload
}
```

Основные команды:

- `EnsureLoaded(snapshot)`;
- `UnloadAndExportSnapshot(reason)`;
- `RemoveReactor(reason)`;
- `Tick(dtSeconds, budget)`;
- `InsertItem(portId, itemId, count)`;
- `InsertFluid(portId, fluidId, amount)`;
- `DrainOutput(portId, requestedAmount)`;
- `ChangePortSettings(portId, settings)`;
- `RequestSnapshot(reason)`;
- `RequestMetrics(mask)`.

Команды, которые меняют состояние, должны проверять `expectedSnapshotVersion`. Если версия не совпадает, Rust возвращает отказ, а не применяет команду к неизвестному состоянию.

## Отчёты

Каждый отчёт должен иметь:

```text
ReactorReport {
  reportId
  commandId?
  structureId
  snapshotVersion
  payload
}
```

Основные отчёты:

- `CommandAccepted`;
- `CommandRejected(reason)`;
- `TickCompleted(metrics)`;
- `SnapshotReady(snapshot)`;
- `PortInputAccepted(portId, itemId, acceptedCount)`;
- `PortInputRejected(portId, reason)`;
- `PortOutputAccepted(portId, itemId, extractedCount)`;
- `PortOutputRejected(portId, reason)`;
- `DiagnosticsUpdated(entries)`;
- `ReactorFailed(error)`.

Отчёт о порте не является вторым источником истины и не должен сам менять предметный инвентарь. Он только сообщает результат короткого цикла обмена: сколько предметов можно убрать из входного порта, сколько предметов можно положить в выходной порт, либо почему перенос не выполнен.

Для обычного обмена не нужен долговечный журнал подтверждений. Если команда выполняется синхронно коротким вызовом, результат применяется сразу в рамках того же цикла. Если команда уходит в асинхронную очередь, JVM может держать маленькую карту `commandId -> ожидаемый перенос` только в ОЗУ, чтобы применить ответ к правильному порту и предмету. Эту карту не надо сохранять в мир: при остановке сервера ожидающие переносы отменяются, а предметы остаются в портовых инвентарях.

## Остановка сервера и частичные команды

Нормальный порядок:

```text
1. JVM прекращает принимать новые внешние операции.
2. JVM отправляет RequestSnapshot или UnloadAndExportSnapshot.
3. JVM ждёт отчёт ограниченное время.
4. Если отчёт пришёл, сохраняет новый подтверждённый снимок.
5. Если отчёт не пришёл, сохраняет последний подтверждённый снимок и внешние буферы.
6. JVM вызывает nativeShutdownReactorWorker().
```

Если сервер остановился между отправкой команды и отчётом асинхронного переноса:

- предметы не считаются списанными, пока нет отчёта;
- жидкости не считаются принятыми, пока нет отчёта;
- настройки порта считаются применёнными только после отчёта;
- при следующей загрузке команды создаются заново из сохранённого состояния, а не слепо повторяются из старой очереди.

Команды являются транспортом, а не источником истины. Источник истины: подтверждённый снимок, внешние буферы портов и настройки портов.

## Цикл обмена портов

Основной путь обмена должен быть одним коротким циклом на структуру, а не набором произвольных вызовов портов:

```text
1. Выходы:
   - для каждого настроенного выхода посчитать, сколько предметов помещается в порт;
   - запросить у Rust не больше этого количества;
   - положить в порт ровно количество из PortOutputAccepted;
   - всё, что не поместилось, остаётся не сконвертированным в Rust.

2. Входы:
   - для каждого настроенного входа взять доступный предметный стак;
   - запросить у Rust приём доступного количества;
   - удалить из порта ровно acceptedCount из PortInputAccepted;
   - всё, что Rust не принял, остаётся в портовом инвентаре.
```

Выходы всегда идут раньше входов. Это правило не должно зависеть от вызывающего кода: общий слой должен иметь метод уровня `ReactorPortTransferCycle`, который строит команды в этом порядке и заранее проверяет структуру, порты и вместимость очереди. Низкоуровневые команды `InsertItem` и `ExtractItem` остаются транспортом, а не рекомендуемым внешним интерфейсом для игрового слоя.

Отказ не является успехом. `PortInputRejected` и `PortOutputRejected` должны доходить до слоя портов как отказ, а не превращаться в `Completed`. Состояние структуры может при этом успешно принять сам отчёт, но это отдельный факт и его нельзя смешивать с результатом переноса предметов.

## Предметные и жидкостные порты

Порт может иметь локальный маленький буфер, но этот буфер не является химическим состоянием реактора.

Правило входа предмета:

```text
предмет попал в порт
порт проверил принадлежность структуре
цикл обмена отправил InsertItem
Rust вернул PortInputAccepted(acceptedCount)
JVM списала из порта ровно acceptedCount
```

Если Rust не подтвердил принятие, предмет остаётся в портовом инвентаре. Если инвентарь полон, вход новых предметов отклоняется сразу.

Правило выхода:

```text
JVM посчитала свободное место в выходном порте
цикл обмена отправил ExtractItem(maxItemCount = свободное место)
Rust вернул PortOutputAccepted(extractedCount)
JVM положила в порт ровно extractedCount
если места нет, ExtractItem не отправляется
```

Нельзя терять вещество, если предметный выход забит.

## Частота работы

### Серверный поток

Каждый тик:

- применить ограниченное число готовых отчётов;
- обновить открытые экраны, если пришли новые данные;
- собрать изменения портов в лёгкие очереди.

Нельзя выполнять тяжёлый расчёт химии в серверном потоке.

### Координатор JVM

Раз в 5-20 тиков или по событию:

- собрать пачку команд;
- отправить её через JNI;
- забрать отчёты, если они уже готовы.

Отдельный поток ожидания отчётов может выполнять долгий опрос с небольшим временем ожидания. Он не трогает мир.

### Rust

Rust-исполнитель:

- принимает пачки команд;
- применяет команды последовательно по каждому реактору;
- считает реакторы в своём потоке;
- пишет отчёты в очередь.

В первой версии достаточно одного рабочего потока. Распараллеливание по реакторам можно добавить после стабилизации протокола.

## Динамический каталог веществ

Реакторный исполнитель должен различать два вида состояния:

```text
ReactorState
  зоны
  смеси
  порты
  выходные буферы

CatalogState
  статический каталог
  динамические вещества
  пересобираемый слой динамических реакций
  кислотно-основные спецификации
  спецификации осаждения
  спецификации комплексов
  индексы реакций
```

`ReactorSnapshot` не должен копировать весь статический каталог. Но он должен ссылаться на версию базового каталога и версию динамического расширения.

Рекомендуемое разделение:

- базовый каталог Destroy создаётся один раз и проверяется как неизменяемый;
- динамический каталог принадлежит нативному состоянию мира, а не отдельному реактору;
- несколько реакторов одного мира используют один и тот же динамический каталог;
- снимок мира сохраняет динамическое расширение каталога отдельно от снимков реакторов;
- снимок реактора хранит ссылки на вещества и реакции, плюс проверочную версию каталога.

При восстановлении порядок такой:

```text
1. восстановить базовый каталог Destroy;
2. восстановить динамическое расширение каталога мира;
3. проверить версии и канонические коды динамических веществ;
4. восстановить реакторы из снимков;
5. только после этого принимать команды портов и тики.
```

Если снимок реактора ссылается на вещество, которого нет ни в статическом каталоге, ни в восстановленном динамическом расширении, восстановление должно завершиться явной ошибкой. Нельзя молча выкидывать вещество из смеси или заменять его заглушкой.

### Состав снимка динамического каталога

Динамический каталог может стать большим, поэтому на стороне JVM он не должен раскладываться в человекочитаемую модель веществ и реакций.

JVM хранит динамический каталог как непрозрачные сжатые двоичные данные:

```text
DynamicCatalogStorageRef {
  schemaVersion
  baseCatalogVersion
  catalogVersion
  modelVersion
  contentHash
  compressedByteSize
  uncompressedByteSize
  checkpointStorageKey
  deltaStorageKeys[]
}
```

`DynamicCatalogStorageRef` является частным случаем `NativeBlobRef` с поддержкой цепочки `checkpoint + delta[]`.

В обычном мировом сохранении должны лежать только ссылка, версии, размеры и хэш. Сами данные каталога хранить отдельными сжатыми файлами в папке сохранения мира, например:

```text
data/create_thermodynamics/catalog/
  checkpoint_000042.bin.zst
  delta_000043.bin.zst
  delta_000044.bin.zst
```

NBT не должен содержать полный динамический каталог как огромный массив байтов, если каталог уже вынесен в отдельные файлы. Это уменьшает лишнее копирование, ускоряет сохранение мира и не заставляет JVM держать всю химию в управляемой памяти.

Внутри двоичного формата Rust сохраняет компактные исходные данные, достаточные для проверки и восстановления:

- динамические вещества: идентификатор, канонический графовый код, свойства вещества, теги и признаки запрета для обычной смеси;
- динамические кислотно-основные, осадительные и комплексообразующие спецификации;
- версия модели генерации, версия базового каталога и проверочная сводка.

Динамические реакции и обработанные ключи генерации не входят в снимок каталога. После загрузки они пересобираются по необходимости из восстановленных динамических веществ, статического каталога и текущего пространства генерации. Если когда-нибудь понадобится сохранять крупное реакционное пространство как отдельный снимок планировщика, это должен быть отдельный тип `NativeBlob`, а не часть каталога и не список идентификаторов реакций.

Не сохранять как долговременное состояние:

- индексы `dynamic_substance_id_to_index` и `dynamic_reaction_id_to_index`;
- индексы реакций по веществам;
- индексы реакционных центров;
- карты канонического кода, если они однозначно пересобираются из сохранённых веществ;
- временные очереди генерации;
- расчётные кэши, которые можно безопасно восстановить.

Если какой-то индекс окажется слишком дорогим для пересборки, его можно сохранить отдельно только как проверяемый ускоритель: при загрузке он обязан сверяться с исходными данными и отбрасываться при несоответствии. Нельзя делать такой индекс единственным источником истины.

Запись каталога должна быть устроена как редкие полные снимки и последовательность изменений:

```text
checkpoint 42
delta 43
delta 44
delta 45
```

Когда число или размер изменений превышает предел, Rust создаёт новый полный снимок:

```text
checkpoint 45
```

Старые изменения можно удалить только после успешной записи нового полного снимка и проверки его хэша. JVM не должна сама объединять изменения каталога: она только хранит файлы и передаёт байты в Rust.

### События каталога

Нативный слой должен уметь сообщать JVM о расширении динамического каталога:

```text
CatalogExtended {
  catalogVersion
  addedSubstances[]
  addedReactions[]
  addedSpecs[]
}
```

JVM не использует эти данные для расчёта химии. Она сохраняет их и может показывать в интерфейсе. Расчёт остаётся на Rust-стороне.

### Команды каталога

Нужны отдельные команды:

- `ResolveFrowns(code)`;
- `ResolveStructure(canonicalGraphCode)`;
- `GenerateReactionsFor(substanceId, depth)`;
- `ExportCatalogSnapshot`;
- `ImportCatalogSnapshot`.

Эти команды должны идти через тот же протокол команд и отчётов, что и реакторные операции. Они не должны вызываться напрямую из клиентского интерфейса без серверной проверки.

## Риски памяти и производительности

Эта подсистема легко станет дорогой, если реализовать её как копирование больших снимков между слоями. Основные ограничения должны быть частью архитектуры с первой реализации.

### Каталог

- Динамический каталог должен быть один на мир, а не один на реактор.
- Реакторы хранят только версию каталога и ссылки на вещества.
- Расширение каталога передаётся как изменение: добавленные вещества, реакции и спецификации, а не полный каталог.
- JVM не должна держать отдельную полную копию графов всех веществ для каждого открытого экрана.
- JVM не должна иметь человекочитаемую объектную модель динамического каталога. Для сохранения ей достаточно ссылки на сжатые файлы, версий, размеров и хэшей.
- Сжатый двоичный снимок каталога не должен распаковываться в JVM ради сохранения или загрузки. Распаковка и проверка принадлежат Rust.
- При открытом интерфейсе JVM запрашивает у Rust только нужную страницу или сводку каталога, а не весь каталог.
- При восстановлении мира сначала восстанавливается каталог, затем реакторы. Обратный порядок запрещён.

### Снимки реакторов

- Полный `ReactorSnapshot` нужен при сохранении мира, выгрузке структуры и ручном запросе диагностики.
- Обычный тик не должен создавать полный снимок.
- Тиковый отчёт должен содержать только версию, краткие метрики и изменения, которые реально нужны JVM.
- Для экрана контроллера отправлять данные только открытой вкладки и только при изменении версии соответствующего раздела.
- Историю температур, давлений и скоростей хранить как ограниченный кольцевой буфер с прореживанием, а не как список всех тиков.

### Очереди команд и отчётов

Все очереди должны иметь фиксированные ограничения:

```text
commandOutboxCapacity
reportInboxCapacity
nativeReportQueueCapacity
uiHistoryCapacity
snapshotMaxBytes
catalogDeltaMaxEntries
```

Переполнение очереди является явным состоянием отказа или обратного давления, а не поводом бесконечно выделять память. Допустимые реакции:

- отказать новой внешней операции до освобождения места;
- объединить несколько метрик одного реактора в последнюю метрику;
- запросить полный снимок после потери необязательных диагностических отчётов;
- остановить нативную сессию с явной ошибкой, если потеря отчёта нарушает состояние.

Нельзя отбрасывать отчёты, которые подтверждают списание предметов, принятие жидкости, изменение смеси или выгрузку снимка.

### Формат обмена

- В горячем пути не передавать строки веществ в каждой записи. Использовать словарь снимка или компактные номера внутри одной пачки, привязанные к версии каталога.
- Строковые `SubstanceId` допустимы в сохранении, диагностике и редких командах.
- В Rust можно использовать внутренние индексы для скорости, но они не должны попадать в долговременное сохранение как источник истины.
- Общий участок памяти вне управляемой кучи допустим только как фиксированный кольцевой буфер с версией протокола, пределом размера и явной ошибкой переполнения.
- Большие нативные данные передаются как `NativeBlobRef` или поток байтов с проверкой хэша, а не как списки Kotlin-объектов.

### Генерация динамической химии

- Генерацию нельзя запускать от всего каталога как обычный тик реактора.
- Запросы генерации должны быть привязаны к веществам конкретной смеси, явному запросу игрока или планировщику.
- Уже обработанные ключи генерации считаются внутренним ускорителем текущего нативного экземпляра. В обычный снимок каталога они не входят; после загрузки генерация должна корректно и детерминированно дойти до нужного состояния заново.
- Аварийные пределы генерации остаются только проверкой нарушения алгоритма, а не нормальным способом завершения.

### Потоки

- Серверный поток применяет ограниченное число отчётов за тик.
- Нативный поток не держит ссылки на объекты Minecraft и не вызывает JVM-код мира.
- Поток ожидания отчётов может ждать нативу, но обязан завершаться при выгрузке мира и остановке сервера.
- Если отчётов накопилось слишком много, серверный поток применяет их порциями и сохраняет явный признак отставания для интерфейса.

## Пользовательский интерфейс контроллера

Контроллер должен стать полноценной панелью управления, а не одним экраном со всеми числами.

Вкладки:

- `Overview`: состояние структуры, нативной связи, ошибки, температура, давление, объём.
- `Zones`: зоны, фазы, состав, газовый зазор, давление, температура.
- `Ports`: режимы портов, направления, фильтры, привязки предметов и веществ.
- `Mixture`: таблица веществ, количество, концентрации, фазы.
- `Reactions`: текущие реакции, скорости, тепловые эффекты, ограничивающие реагенты.
- `Diagnostics`: ошибки сборки, незагруженные чанки, отказы нативы, последние команды.
- `History`: сжатые графики температуры, давления, состава и скорости реакций.

Клиент не пересчитывает химию. Он получает только снимки отображения:

```text
ReactorControllerViewSnapshot
ReactorZoneViewSnapshot
ReactorPortViewSnapshot
ReactorMixtureViewSnapshot
ReactorReactionViewSnapshot
ReactorDiagnosticsSnapshot
```

Сервер отправляет конкретному игроку только то, что он сейчас наблюдает. Подробная вкладка состава не должна обновляться, если игрок смотрит только обзор.

## Проверки

### Чистые JVM-тесты

- структура создаёт `EnsureLoaded`;
- удаление структуры создаёт `UnloadAndExportSnapshot` или `RemoveReactor`;
- незагруженный чанк переводит реактор в `SUSPENDED_UNKNOWN`, а не удаляет состояние;
- цикл обмена портов ставит выходные команды перед входными;
- предмет не списывается до `PortInputAccepted`;
- отказ Rust оставляет предмет в портовом инвентаре;
- выходной порт запрашивает у Rust не больше свободного места;
- `PortInputRejected` и `PortOutputRejected` видны вызывающему слою как отказ переноса, а не как общий `Completed`;
- повторная загрузка создаёт нативный экземпляр из последнего подтверждённого снимка;
- старый отчёт с устаревшей версией не применяется;
- переполнение очереди команд даёт типизированный отказ, а не рост памяти;
- экран контроллера получает только данные открытой вкладки;
- две структуры одного мира ссылаются на один каталоговый снимок, а не хранят две копии каталога;
- сохранение мира хранит крупные нативные данные через `NativeBlobRef`, а не через списки JVM-объектов реакций, каналов или графов.

### JNI-тесты

- повреждённая команда возвращает явный отказ;
- команда с неверной версией снимка не применяется;
- `nativePollReactorReports(timeout)` просыпается по отчёту;
- `nativeWakeReactorWorker()` прерывает ожидание;
- `nativeShutdownReactorWorker()` завершает рабочий поток без зависания;
- повторный `EnsureLoaded` для того же `structureId` идемпотентен.

### Rust-тесты

- экспорт и импорт снимка сохраняют смесь и порты;
- экспорт и импорт сохраняют режим объёма зоны, герметичность, прошедшее время, переходы, входы и выходы;
- экспорт и импорт сохраняют полный фазовый состав смеси: водную, органическую, расплавленную, газовую, сверхкритическую и твёрдую части;
- тик после восстановления даёт тот же результат, что тик без выгрузки;
- выходные буферы не теряют дробные количества;
- остановка после части команд не повреждает состояние worker;
- неизвестный реактор, неизвестный порт и неверный `commandId` возвращают явную ошибку;
- реактор со ссылкой на динамическое вещество не восстанавливается, пока не восстановлен динамический каталог;
- индексы веществ пересобираются из стабильных идентификаторов и канонических графов;
- реакции для восстановленных динамических веществ пересобираются по необходимости, а не читаются из сохранённого списка;
- переполнение нативной очереди отчётов возвращает явную ошибку или обратное давление.

### Тесты снимка каталога

- динамическое вещество восстанавливается по каноническому графовому коду и свойствам;
- снимок каталога не содержит идентификаторов динамических реакций;
- после восстановления динамического вещества генерация по этому веществу заново создаёт реакции, ссылающиеся только на существующие вещества;
- кислотно-основные, осадительные и комплексообразующие спецификации сохраняются и проходят проверку ссылок;
- обработанные ключи генерации не сохраняются как состояние каталога;
- временные индексы можно удалить перед сохранением и получить тот же рабочий каталог после загрузки;
- JVM-сохранение динамического каталога содержит только ссылку на сжатые двоичные файлы, версии, размеры и хэши;
- повреждённый хэш снимка или изменения каталога даёт явную ошибку восстановления;
- цепочка `checkpoint + delta[]` восстанавливается в тот же каталог, что и новый полный снимок после уплотнения;
- JVM не создаёт объектную модель всех динамических веществ и реакций при загрузке мира;
- крупные снимки реакционного пространства, если они понадобятся планировщику, сохраняются отдельным нативным двоичным форматом, а не Kotlin-структурами и не частью каталога.

### Игровые проверки

- собрать простой реактор, положить связанный предмет во входной порт, увидеть принятие вещества;
- перезагрузить мир и убедиться, что состояние реактора восстановилось;
- выгрузить чанк с частью структуры и убедиться, что состояние стало `UNKNOWN`, а не `NOT_FORMED`;
- сломать камеру и убедиться, что нативный экземпляр удалён или выгружен через снимок;
- открыть экран контроллера и убедиться, что подробные данные приходят только для открытой вкладки.

## Порядок реализации

Главная проверочная цель первой половины работ: каталог восстановился, реактор восстановился, тик после восстановления даёт тот же результат, что тик без выгрузки. Порты и интерфейс подключаются только после этого.

1. Зафиксировать текущее состояние: структура собирается, но нативная связь ещё не является рабочим источником химии.
2. Зафиксировать текущую нативную структуру: статический каталог, `DynamicChemistryRegistry`, `ReactorsWorker`, привязки предметов, текущие поля `Reactor`, `ReactorZone` и `Mixture`.
3. Реализовать общий нативный двоичный формат `NativeBlob`: заголовок, тип, версии, размеры, сжатие, хэш, секции и проверку повреждённых данных.
4. Реализовать Rust-снимок динамического каталога: полный снимок, изменение, восстановление, проверку ссылок, восстановление индексов и отказ при несовпадении хэша.
5. Реализовать Rust-снимок реактора: зоны, смеси, фазы, переходы, входы, выходы, выходные буферы и постоянный контекст реакции.
6. Добавить Rust-тесты `export -> import -> tick`, где восстановленный реактор даёт тот же результат, что реактор без выгрузки.
7. Добавить JVM-хранилище нативных двоичных снимков: `NativeBlobRef`, отдельные сжатые файлы, безопасную запись через временный файл, проверку размера и хэша.
8. Добавить ограничения размеров: очереди команд, очереди отчётов, снимков, изменений каталога, истории интерфейса и размера одного нативного двоичного куска.
9. Добавить JNI-команды снимков: `nativeExportCatalogSnapshot`, `nativeImportCatalogSnapshot`, `nativeExportReactorSnapshot`, `nativeCreateReactorFromSnapshot`.
10. Добавить протокол команд и отчётов: `submit`, `poll`, `wake`, `shutdown`, версии снимков, отказ при устаревшей версии и отказ при переполнении очереди.
11. Реализовать `ReactorWorldRuntime`, который объединяет структуру мира, ссылки на нативные снимки, каталог и активную нативную сессию.
12. Подключить жизненный цикл структуры: сборка, изменение, выгрузка, загрузка, удаление, `SUSPENDED_UNKNOWN` для незагруженных чанков.
13. Подключить простой активный реактор без периферии: создание нативного экземпляра, тик, запрос кратких метрик, экспорт снимка при выгрузке.
14. Подключить серверный обработчик тика: ограниченное применение отчётов, отсутствие химических расчётов в серверном потоке.
15. Подключить предметный входной порт через подтверждённые отчёты: буфер порта, команда, подтверждение Rust, списание предмета только после подтверждения.
16. Подключить выходные химические буферы и предметный выход: если предмет не помещается, вещество остаётся в химическом буфере.
17. Подключить жидкостные порты: объём, температура, фазы, частичное принятие и отказ без потери вещества.
18. Добавить сетевые снимки для экрана контроллера: сначала обзор, затем вкладки зон, смеси, реакций, портов, диагностики и истории.
19. Добавить уплотнение каталога: цепочка `checkpoint + delta[]` превращается в новый проверенный `checkpoint`.
20. После стабилизации измерить стоимость JNI-пачек и только по результатам решать, нужен ли общий участок памяти.

## Запрещённые решения

- Не хранить химическое состояние в нескольких сущностях блоков.
- Не считать нативный экземпляр единственным долговременным состоянием.
- Не восстанавливать реактор без восстановления динамического каталога, если в смеси есть динамические вещества.
- Не сохранять внутренние индексы веществ, реакций, фаз и сайтов как долговременную истину.
- Не сохранять смесь без разбиения по фазам.
- Не держать полный динамический каталог внутри каждого реактора.
- Не раскладывать динамический каталог в человекочитаемые JVM-структуры веществ и реакций ради сохранения.
- Не раскладывать реакции, каналы, графы, реакционное пространство или полный снимок реактора в JVM-структуры, если JVM не должна принимать решения по их содержимому.
- Не хранить большой динамический каталог целиком в NBT, если он вынесен в отдельные сжатые файлы.
- Не распаковывать сжатый снимок каталога в JVM, если операция должна только сохранить, загрузить или передать его в Rust.
- Не создавать полный снимок реактора на каждый тик.
- Не делать неограниченные очереди команд, отчётов, истории интерфейса или изменений каталога.
- Не менять мир Minecraft из нативного потока.
- Не вызывать долгий опрос в серверном потоке.
- Не списывать предмет до отчёта Rust.
- Не считать незагруженный чанк сломанной структурой.
- Не делать клиентские пересчёты химии.
- Не рассылать игроку подробные данные вкладок, которые он не открыл.
- Не добавлять общий участок памяти до стабилизации формата команд и отчётов.
