# Общая проверяемая система интерфейсов

## Кратко

Нужно сделать общий слой интерфейсов для мода, а не отдельный экран реактора с ручными координатами. Интерфейс должен описываться смысловыми узлами, проходить расчёт раскладки, проверяться до запуска игры и отрисовываться одним и тем же набором команд как в Minecraft, так и в предварительном просмотре вне игры.

Главная цель: агент или разработчик должен видеть ошибки раскладки через тесты, отчёт и картинки предварительного просмотра, а не через ручной запуск Minecraft и визуальную подгонку.

## Принципы

- Экран Minecraft не является источником истины. Он только показывает серверный снимок состояния и отправляет команды игрока.
- Нельзя рисовать текст по свободным координатам. Любой текст принадлежит прямоугольной области.
- Нельзя добавлять ручные числа координат в конкретные экраны. Размеры, отступы и цвета задаются темой и расчётом раскладки.
- Нельзя иметь разные пути для игры и предварительного просмотра. Оба пути должны получать один и тот же список команд отрисовки.
- Нельзя скрыто использовать старые решения ради совместимости. Старые ручные экраны должны быть перенесены на новый слой или оставаться явно перечисленными как ещё не перенесённые.
- Переполнение текста не должно кидать ошибку в игровом рантайме. Для каждого динамического текста должна быть заранее задана политика: обрезка, многоточие, перенос, прокрутка, подсказка или отказ на этапе проверки фиксированных образцов.
- Ошибки вроде выхода текста за границы должны ловиться в тестах, предварительном просмотре и проверочных сценариях. В игре они не должны ломать экран, но должны быть невозможны для штатных экранов после проверок.
- Если строка переменной длины может стать слишком длинной, это не исключение в игре, а часть контракта узла: например `EllipsizeWithTooltip`, `WrapLines(maxLines = 2)` или `ScrollableText`.
- Интерфейс не должен пересчитывать химию, структуру мультиблока, фильтры, порты или состояние нативного слоя.

## Архитектура

Поток данных:

```text
серверный снимок состояния
  -> описание интерфейса
  -> расчёт раскладки
  -> проверка ограничений
  -> команды отрисовки
  -> исполнитель Minecraft
  -> исполнитель предварительного просмотра
```

Ключевой результат расчёта - не пиксели, нарисованные сразу в `GuiGraphics`, а список команд:

```text
DrawPanel(...)
DrawFrame(...)
DrawText(...)
DrawIcon(...)
DrawItem(...)
DrawTableRow(...)
DrawTooltipAnchor(...)
```

Minecraft-экран и предварительный просмотр должны исполнять эти команды одинаково.

## Типы ядра

Геометрия:

```kotlin
data class UiRect(
    val x: Int,
    val y: Int,
    val width: Int,
    val height: Int,
) {
    val right: Int get() = x + width
    val bottom: Int get() = y + height

    fun contains(child: UiRect): Boolean =
        child.x >= x && child.y >= y && child.right <= right && child.bottom <= bottom

    fun intersects(other: UiRect): Boolean =
        x < other.right && right > other.x && y < other.bottom && bottom > other.y
}
```

Политика текста:

```kotlin
sealed interface TextOverflowPolicy {
    data object FailInValidation : TextOverflowPolicy
    data object Clip : TextOverflowPolicy
    data object Ellipsize : TextOverflowPolicy
    data object EllipsizeWithTooltip : TextOverflowPolicy
    data class WrapLines(val maxLines: Int) : TextOverflowPolicy
    data object HorizontalScroll : TextOverflowPolicy
}
```

Важно: `FailInValidation` не означает падение игрового рантайма. Это требование к проверочным сценариям. В игре такой текст должен либо не встречаться, либо быть заменён на безопасную политику.

Измеритель текста:

```kotlin
interface UiTextMeasurer {
    fun width(text: String): Int
    fun lineHeight(): Int
}
```

Диагностика раскладки:

```kotlin
sealed interface UiLayoutDiagnostic {
    val nodeId: String

    data class TextWouldOverflow(
        override val nodeId: String,
        val text: String,
        val rect: UiRect,
        val textWidth: Int,
        val policy: TextOverflowPolicy,
    ) : UiLayoutDiagnostic

    data class NodeOutsideParent(
        override val nodeId: String,
        val parent: UiRect,
        val child: UiRect,
    ) : UiLayoutDiagnostic

    data class UnexpectedOverlap(
        override val nodeId: String,
        val first: UiRect,
        val second: UiRect,
    ) : UiLayoutDiagnostic
}
```

Результат раскладки:

```kotlin
data class UiLayoutResult(
    val commands: List<UiDrawCommand>,
    val diagnostics: List<UiLayoutDiagnostic>,
) {
    fun requireValidationClean() {
        require(diagnostics.isEmpty()) {
            diagnostics.joinToString(separator = "\n")
        }
    }
}
```

Эту проверку можно использовать в тестах и предварительном просмотре. Игровой экран не должен вызывать `requireValidationClean()` во время обычной отрисовки.

## Политика ошибок и переменных строк

Для строк переменной длины запрещено полагаться на то, что "обычно помещается".

Каждый динамический текст должен объявить одну из политик:

- `EllipsizeWithTooltip` для названий веществ, предметов, портов, структур.
- `WrapLines(maxLines = N)` для диагностик и описаний.
- `HorizontalScroll` для редких случаев, где важна полная строка и есть отдельная область прокрутки.
- `Clip` только для декоративных или незначимых подписей, где потеря хвоста не ломает смысл.
- `FailInValidation` только для фиксированных подписей интерфейса, которые обязаны помещаться всегда.

Примеры:

```kotlin
text(
    id = "mixture.row.substance",
    value = substanceName,
    overflow = TextOverflowPolicy.EllipsizeWithTooltip,
)
```

```kotlin
text(
    id = "diagnostic.message",
    value = diagnosticText,
    overflow = TextOverflowPolicy.WrapLines(maxLines = 3),
)
```

```kotlin
text(
    id = "tab.overview",
    value = "Overview",
    overflow = TextOverflowPolicy.FailInValidation,
)
```

В игровом рантайме результатом переполнения не должна быть ошибка. Результатом должна быть заранее выбранная отрисовка.

## Компоненты

Минимальный набор компонентов:

- `UiPanel` - прямоугольная область с рамкой и фоном.
- `UiRow` - горизонтальное размещение детей.
- `UiColumn` - вертикальное размещение детей.
- `UiTabs` - вкладки с выбранным состоянием.
- `UiText` - текст с политикой переполнения.
- `UiMetricCard` - заголовок, значение, вторичная строка.
- `UiTable` - таблица с фиксированными или весовыми столбцами.
- `UiDiagnosticBox` - блок сообщений с переносом строк.
- `UiItemSlotGrid` - сетка настоящих слотов, если экран действительно содержит инвентарь.

Компоненты не должны знать конкретные сущности реактора. Реактор, порт и другие экраны строятся поверх этих компонентов.

## Предварительный просмотр

Предварительный просмотр должен быть обязательным инструментом разработки интерфейсов.

Команда:

```text
./gradlew :modules:ui-preview:render
```

Результат:

```text
build/ui-preview/
  index.html
  layout-report.json
  reactor-controller-overview-active.png
  reactor-controller-overview-empty.png
  reactor-controller-zones-many.png
  reactor-controller-mixture-long-names.png
```

Требования:

- Использовать те же описания интерфейса, что и Minecraft.
- Использовать те же команды отрисовки.
- Использовать те же текстуры фона и рамок из ресурсов мода.
- Использовать измерение текста, максимально близкое к Minecraft.
- Писать отчёт с диагностикой по каждому сценарию.

Если точное воспроизведение шрифта Minecraft сразу окажется слишком дорогим, первый слой допускает проверочный измеритель, который завышает ширину текста. Это безопаснее, чем занижать ширину и получать наложения в игре. Но итоговая цель - использовать реальные данные ширины символов Minecraft.

## Проверочные сценарии

Для каждого экрана нужно хранить набор сценариев.

Для будущего экрана реактора:

- структура не собрана;
- структура собрана, нативный реактор ещё не создан;
- активный реактор с одной зоной;
- много зон;
- пустая смесь;
- смесь с длинными названиями веществ;
- смесь с большими числами концентраций;
- высокое давление;
- длинная диагностика отказа;
- порты с фильтрами и ошибками передачи.

Для каждого сценария проверять:

- нет недопустимых диагностик раскладки;
- фиксированные подписи помещаются;
- динамические строки имеют явную политику переполнения;
- команды отрисовки не выходят за размер окна;
- таблицы показывают только видимые строки;
- подсказки создаются для обрезанных важных строк.

## Перенос текущих экранов

Переносить нужно по одному экрану.

Порядок:

1. Экран контроллера реактора.
2. Экран предметного порта.
3. Будущие экраны жидкостного порта.
4. Будущие экраны диагностики и планировщика.

Для каждого экрана:

- сначала создать снимки состояний;
- затем создать описание интерфейса;
- затем добавить проверочные сценарии;
- затем подключить Minecraft-исполнитель;
- затем убрать ручную отрисовку текста из старого экрана.

## Запреты для новых экранов

Нельзя:

- вызывать `drawString` напрямую из конкретного экрана, кроме временно перечисленных старых экранов;
- хранить координаты элементов внутри экрана;
- рисовать текст без `TextOverflowPolicy`;
- смешивать реальное хранилище и настройку;
- вычислять серверное состояние на клиенте;
- иметь отдельную логику предварительного просмотра, отличную от игровой;
- добавлять экран без проверочных сценариев.

Допустимо:

- низкоуровневому исполнителю вызывать `GuiGraphics`;
- компоненту `UiItemSlotGrid` знать про прямоугольники слотов;
- экрану хранить выбранную вкладку и положение прокрутки;
- экрану отправлять типизированные команды на сервер.

## План реализации

### Этап 1. Ядро геометрии и текста

- Создать типы `UiRect`, `UiSize`, `UiInsets`, `UiAlignment`.
- Создать `UiTextMeasurer`.
- Создать `TextOverflowPolicy`.
- Добавить тесты на пересечения, вложенность, размеры текста и политики переполнения.

Проверка:

```text
./gradlew :modules:v1_21_1:v1_21_1-neoforge:test --tests "*UiGeometry*"
```

Если тесты loader-модуля слишком тяжёлые, перед этим создать `modules/ui-core` и перенести туда ядро.

### Этап 2. Узлы интерфейса и расчёт раскладки

- Создать `UiNode`.
- Добавить `UiText`, `UiPanel`, `UiRow`, `UiColumn`.
- Создать `UiLayoutEngine`.
- Создать `UiLayoutDiagnostic`.
- Добавить тесты, где текст намеренно не помещается.
- Проверить, что в тестах это диагностируется, а не маскируется.

### Этап 3. Команды отрисовки

- Создать `UiDrawCommand`.
- Создать `MinecraftUiRenderer`.
- Экран Minecraft пока не переносить.
- Добавить тест, что одно и то же дерево интерфейса даёт стабильный список команд.

### Этап 4. Тема Create-подобного интерфейса

- Создать `CreateUiTheme`.
- Вынести туда цвета, отступы, размеры строк, высоты вкладок.
- Запретить новым компонентам хранить свои произвольные цвета и отступы без темы.

### Этап 5. Таблицы и вкладки

- Добавить `UiTabs`.
- Добавить `UiTable`.
- Добавить прокрутку видимой области таблицы.
- Добавить тесты на длинные строки, много строк и узкие столбцы.

### Этап 6. Предварительный просмотр вне игры

- Создать модуль `modules/ui-preview`.
- Реализовать `PngUiRenderer`.
- Научить его исполнять `UiDrawCommand`.
- Подключить ресурсы фона и рамок.
- Добавить `layout-report.json`.
- Добавить сценарии для первого экрана.

Проверка:

```text
./gradlew :modules:ui-preview:render
```

### Этап 7. Первый перенос: экран контроллера реактора

- Создать построитель интерфейса контроллера, например `ReactorControllerUi.kt`.
- Он должен возвращать дерево `UiNode`, а не рисовать сам.
- Сделать вкладки:
  - `Overview`;
  - `Zones`;
  - `Mixture`;
  - `Ports`;
  - `Diagnostics`.
- `ReactorControllerScreen` должен только:
  - получить состояние;
  - выбрать вкладку;
  - вызвать построитель;
  - рассчитать раскладку;
  - отдать команды исполнителю.

### Этап 8. Убрать ручную отрисовку из перенесённого экрана

- В `ReactorControllerScreen` не должно остаться прямого рисования текста.
- Разрешён только вызов общего исполнителя.
- Добавить поисковую проверку на прямые вызовы `drawString` в перенесённом экране.

### Этап 9. Расширить предварительный просмотр

- Добавить все крайние сценарии контроллера.
- Сохранять картинки по вкладкам.
- Делать отчёт удобным для агента: файл, экран, вкладка, узел, проблема.

### Этап 10. Перенести остальные экраны

- Предметный порт.
- Жидкостный порт, когда он появится.
- Будущие экраны аналитики, диагностики и планировщика.

## Критерии готовности

Система считается готовой для первого использования, когда:

- экран контроллера реактора перенесён на новый слой;
- предварительный просмотр создаёт PNG по нескольким состояниям;
- тесты ловят переполнение фиксированных подписей;
- динамические строки имеют явные политики переполнения;
- в игровом рантайме нет исключений из-за длинных строк;
- Minecraft-экран и предварительный просмотр используют один и тот же список команд отрисовки;
- прямые ручные координаты убраны из перенесённого экрана.

## Почему это должно сократить итерации

Сейчас ошибка видна только после запуска Minecraft. После внедрения:

- агент запускает тесты раскладки;
- агент запускает предварительный просмотр;
- агент получает PNG;
- агент получает машинный отчёт;
- большинство ошибок видно до игры.

Это не просто улучшение внешнего вида. Это изменение процесса: интерфейс становится проверяемой системой, а не набором пиксельных догадок.
